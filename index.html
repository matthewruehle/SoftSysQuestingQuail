Abstract

This file transfer server implements a reduced version of the HTTP protocol so that clients can request files by name using GET requests. This is built on top of a TCP server, which is patterned from code in Unix Network Programming by W. Richard Stevens. We also referenced the code from TOC to better understand the FTP protocol while building our server and client.

Background

The HTTP server/client we made has a similar framework to any general website you would find on the internet. Our clients can use GET requests to ask the server for a specific page, and the server dynamically finds and returns the requested page if it is available. The key difference between our client and a traditional “browser” is the visualization part. As it stands, our client simply receives files and allows the user to browse them at their leisure.

We use the C socket library to allow our server to talk to our client. A socket is a communication channel between two processes, either on the same machine or on different machines that communicate over a network. In order to establish a connection between our server and client, we first bind() the server’s socket to a port, a communication endpoint that represents a specific process on the machine. Next the client has to connect() to this socket, using the socket address (essentially, the IP address) and the port number. The server uses listen() to check for clients trying to connect to it, and can then accept() them: creating a new socket that specifically connects a server/client pair. This new socket is spun off into a child process through fork(), allowing the original socket to keep listening for new connections.

Once a socket between server and client is established, the send() and recv() calls are often used to communicate. In the case of our client, we use the function Readline() instead of recv(), which is similar but retrieves just one line at a time, accepts fewer options, and includes error-checking.

Implementation

Over the course of our project, we refined our end-goal several times. Eventually, we fixed on a series of steps beginning with a TCP server, which we then expanded to an FTP server and client before concluding with an HTTP server/client pair. After these were implemented, we moved on to our “stretch goals” - features like allowing the client to specify the file they wanted, and having the server dynamically parse requested filenames.

We had the server use forked child processes to manage a significant number of simultaneous requests from users without forcing them to wait in queues for service and allowing the server itself to be relatively load free.

Throughout the project, we had several choices to make regarding which protocols to implement, and how low-level we wanted to build our code. For example - did we want to try building system calls from scratch, or concentrate on a higher layer of architecture? How could our team members best meet their learning goals? Ultimately, we decided to keep with the common standard for system calls that set up the connections between client and server; we then dove into understanding and expanding the information exchanged between the server and client.

Our first step was a TCP server, because it’s a fairly simple and well-documented protocol; we wanted to understand the actions which the server was undertaking before adding additional complications like file IO and formatting. We then moved on to a file-transfer server, FTP, both due to the significance of servers in online architecture and so that we could block out specific server and client behaviors, rather than having all processes function as both.

Not everything in the project went smoothly. We had a few circumstances where both our inexperience in C and the relative difficulty of debugging over a network connection combined to keep us from figuring out what we later realized were easy problems. For example: we originally had the server always send the file size in a fixed-size character array, which worked under certain conditions but failed over others; as such, the client would sometimes receive incomplete or garbled messages. We eventually determined that the problem stemmed from the unchanging size of the character array - in particular, from how we defined the arrays. over it. We had assumed that it was already initialized to be filled with null characters, but the array actually held whatever data had last been written there. As such, when we sent a line, we in turn included a lot of “junk” data - a simple enough problem to resolve now, but one which took us quite a while to identify and debug.

Over the course of solving this problem, we realized that our debugging strategy - having four people try to simultaneously debug the code on the same computer - was not as effective as pairs of people. We also learned about new tools that we felt could help us more effectively debug server problems in the future like packet sniffers. Using that, we could have determined exactly what the server sent and come to the realization that the problem was on the server side.

Our other main challenges - a general inexperience with both C and network programming - were addressed through extensive reading on the subject, as well as whole-team code read-throughs. Code read-throughs were when the team put a piece of code that did something similar to what we wanted to do up on a big screen and went through it line by line. The goal, which was usually successful, was to have everyone be on the same page as to what we were doing, the tools we were working with, and to have people fill in other’s knowledge gaps.
